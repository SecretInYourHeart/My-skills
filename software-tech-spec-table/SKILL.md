# 软件产品技术参数表生成

将软件功能说明书、产品白皮书或功能说明文档转化为结构化的技术参数表，用于招标文件或内部文档编制。

## 适用场景

- 招标文件中的技术参数需求表编制
- 内部文档中的软件功能清单整理
- 产品白皮书到功能规格的转化
- 不同软件产品功能点的结构化对比

## 输入要求

用户提供以下任一类型的源文档：
- 软件功能说明书（Word/PDF/文本）
- 产品白皮书
- 功能说明文档（包含功能模块、技术指标、性能参数等）
- 多个软件产品的功能文档（用于生成对比表）

用户需明确输出格式（如未指定，默认输出"带下载按钮的HTML文件"）。

## 输出格式

| 格式 | 说明 | 适用场景 |
|------|------|----------|
| 可直接复制的标准表格 | Markdown 表格，可直接复制粘贴到 Excel | 快速预览、编辑 |
| 带下载按钮的 HTML 文件 | 独立 HTML 文件，内含完整参数表，打开即可用 | 正式文档、归档 |
| 纯表格文本 | 纯文本格式，无 Markdown 语法 | 特殊系统兼容 |

## 列结构（默认）

| 列名 | 说明 |
|------|------|
| 序号 | 全表递增编号 |
| 一级功能域 | 功能的大分类（相同内容自动合并单元格） |
| 二级功能模块 | 功能的子分类 |
| 功能点名称 | 具体功能点的名称 |
| 技术参数与功能要求 | 功能的详细技术描述和要求 |
| 是否必须 | 标识该功能点的强制程度 |
| 备注 | 补充说明（可为空） |

如用户有自定义列结构需求，按用户指定的列结构执行。

## "是否必须"取值规范

按以下优先级使用标准取值：
- **★必须** — 核心功能，不可缺失
- **必须** — 重要功能，应包含
- **应当** — 建议功能，优先包含

当源文档中已有类似标注（如"必选""必要""必备"），统一映射到上述取值。无法从源文档判断时，默认填"必须"。

## 处理流程

### 1. 解析源文档

读取用户提供的文档，识别以下信息：
- 功能模块层级结构（一级域 → 二级模块 → 功能点）
- 各功能点的技术描述和参数要求
- 功能点的优先级/必要性标识（如有）
- 性能指标、兼容性要求等附加信息

### 2. 结构化提取

按列结构逐项填充：
- 从文档结构中提取一级功能域和二级功能模块
- 将功能描述规范化为"功能点名称 + 技术参数与功能要求"
- 根据文档上下文判断"是否必须"等级
- 对模糊信息进行合理推断并标注备注

### 3. 格式化输出

按用户指定的输出格式生成结果：

**标准表格格式**：
- 使用 Markdown 表格语法
- 一级功能域相同的行，仅首行填写内容，其余留空（Excel 粘贴后手动合并）
- 确保列对齐、内容完整

**HTML 文件格式**：
- 生成独立 HTML 文件，使用 `show_widget` 的 preview_url 展示
- HTML 内嵌 CSS，确保表格样式规范
- 一级功能域相同内容使用 `rowspan` 自动合并
- 添加"下载为 Excel"按钮（使用 CSV 导出或纯 HTML 表格格式，确保粘贴到 Excel 可自动对齐）
- 文件编码 UTF-8，适配中文

**纯表格文本格式**：
- 使用制表符或竖线分隔
- 保持列对齐
- 不包含任何 Markdown 格式语法

### 4. 质量检查

输出前逐项检查：
- 每个功能点是否有完整的技术参数描述
- 层级结构是否正确（一级域包含多个二级模块，二级模块包含多个功能点）
- "是否必须"列是否使用标准取值
- 序号是否连续递增
- 表格结构是否清晰，粘贴到 Excel 后是否自动对齐

## 特殊情况处理

- **多产品对比**：当用户提供多个产品文档时，在每个功能点后增加"满足情况"列，逐产品标注"满足/部分满足/不满足/—"
- **缺少技术参数**：如源文档仅有功能名称无详细参数，在"技术参数与功能要求"列中根据功能名称补充合理的通用技术描述，并在"备注"列标注"根据功能名称推断"
- **文档格式不标准**：对非结构化文本，通过语义分析识别功能模块边界，尽量还原层级结构
- **列结构自定义**：用户指定自定义列结构时，优先按用户要求执行；仅在用户明确要求时才使用默认列结构
